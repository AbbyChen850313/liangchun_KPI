## DECISIONS

### 決策 1: 四季評分隔離模型（Quarter-Based Isolation）
- **決策**：使用 Quarter 作為評分作用域單位，不同季度評分互相獨立存儲與計算
- **理由**：HR 年度考績獨立進行，評分標準/評分者可能季度間變化；防止跨季度評分互相覆蓋
- **放棄方案**：全局評分表（無法區分季度）、時間戳版本控制（複雜度高）、完全無狀態（無法追溯）

### 決策 2: 年度加總快照模式（Annual Summary Snapshot）
- **決策**：年度加總是四季評分的只讀快照，由 batch-submit 觸發一次性計算
- **理由**：合規需求（全季度完成後才生成）、防止數據不一致、簡化計算邏輯
- **放棄方案**：實時計算（低效易出錯）、Webhook 觸發器（Sheets 不支援）、分布式鎖（過度設計）

### 決策 3: 評分狀態機（Scoring State Machine）
- **決策**：評分生命周期為 draft → submitted → history 三態，草稿不覆蓋已提交
- **理由**：UX 需求（可多次草稿儲存）、審計追蹤（區分草稿和正式版）、防止誤操作覆蓋
- **放棄方案**：無狀態覆蓋（易誤操作）、版本控制表（複雜度高）、軟刪除（磁盤浪費）

### 決策 4: 員工清單版本化（Employee List Versioning by Quarter）
- **決策**：每季度維護一份員工清單快照，評分時只能評該季度有效員工
- **理由**：應對組織變化（入職/離職）、數據完整性（防止評分不存在的員工）、合規追蹤
- **放棄方案**：全年固定名單（無法應對人事變動）、動態名單（無歷史追蹤）、前端自選（安全風險）

### 決策 5: LIFF 測試環境路由（VITE_IS_TEST 環境變數）
- **決策**：使用 VITE_IS_TEST 環境變數控制 LIFF_ENDPOINT，區分測試/正式 LIFF URL
- **理由**：開發隔離（測試環境不污染正式 token）、快速迭代（自動選擇環境）、安全防護
- **放棄方案**：運行時動態判斷（增加延遲）、本地 mock（無法測試真實流程）、手動切換（容易出錯）

## ISSUES_FOUND

### 級別分布
- **P0（Critical）**：0 個
- **P1（High）**：1 個（員工重複評分）
- **P2（Medium）**：3 個（quarter 參數、annual summary 時序、員工清單同步）
- **P3（Low）**：3 個（草稿覆蓋、狀態界面混亂、歷史記錄驗證）

### 本輪解決
1. **Quarter 參數未傳遞** → 強制驗證，前端補全調用
2. **Annual Summary 計算時序** → 重構流程，先提交再加總
3. **草稿覆蓋已提交評分** → 狀態檢查，403 拒絕
4. **員工清單版本不同步** → 跨季度驗證，check HR_Employees_{quarter}
5. **季度評分狀態界面混亂** → Tabs 改版，清晰展示各季度狀態
6. **歷史記錄四季齊全驗證缺失** → 前置檢查，206 Partial Content 告知缺失
7. **測試/正式環境 LIFF 混淆** → cloudbuild 分支判斷，環境標記

### 未解決（待後續）
- 同季度多次批提交覆蓋（P2）
- 跨季度評分者權限變化（P2）
- **員工重複評分（同評分者）**（P1 — **建議優先處理**）
- 大量並發季度切換（P3）

## 架構健康度檢查

### 一致性
✅ **強**：四季隔離、快照模式、狀態機確保數據一致性
⚠️ **缺陷**：員工重複評分未防護、跨季度權限變化未記錄

### 可測性
✅ **強**：6 項 AC 全部通過、47 個測試覆蓋核心路徑
⚠️ **缺陷**：邊界情況（同評分者重複、並發場景）測試不足

### 可維護性
✅ **強**：清晰的狀態機、季度隔離降低認知複雜度
⚠️ **缺陷**：跨季度驗證分散在多個地方（需提煉通用函數）

### 安全性
✅ **強**：P0-1 跨部門驗證已修復、CSV 注入防護已上線
⚠️ **缺陷**：權限快照缺失、重複評分無唯一鍵約束
